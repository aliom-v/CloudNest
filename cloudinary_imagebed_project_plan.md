# Cloudinary 图床项目计划书

**副标题：** 基于 Next.js + Cloudinary + Vercel 的在线图床与管理平台  
**文档版本：** v1.1  
**最近更新：** 2026-03-25  
**项目目标：** 构建一个界面好看、上传顺滑、与 Cloudinary 官方后台天然同步，并具备管理员删除能力的在线图床项目  
**推荐部署：** Vercel（首选）  
**交付导向：** 先快速上线 MVP，再逐步补齐管理员能力、删除闭环、签名上传与运营能力

## 二、项目摘要

本项目拟建设一个面向个人创作者、小团队和轻量内容运营场景的在线图床平台。平台前端由 Next.js 构建，负责上传、预览、复制链接与管理员管理；图片的真实存储、CDN 分发与媒体资源管理由 Cloudinary 负责。

通过这种“前台自建、后端托管”的模式，可以显著降低对象存储、图片处理、CDN 与媒体管理的自建成本，同时保留较好的产品体验与后续扩展空间。

**核心结论：**

- 该项目不以 PicList 为核心，而是以 Cloudinary 作为真实媒体后端。
- Web 前端是产品入口，承担上传体验、结果展示与轻量管理职责。
- Web 端与 Cloudinary 官方后台不是“双向同步”，而是“同一资源池的两个入口”。
- 删除、签名、管理类能力必须放在服务端，不允许把敏感密钥暴露到浏览器端。

## 三、项目背景与价值

### 3.1 背景

常见图床工具的问题通常不在“能不能上传”，而在以下几个方面：

- 默认界面粗糙，难以形成可长期使用的产品体验。
- 资源管理与访问链接生成割裂，上传后仍需要跳转多个后台。
- 删除、标签、目录、后续运营能力难以整合在一个入口。
- 自建 OSS / S3 / CDN 方案虽然灵活，但初期投入和维护成本偏高。

Cloudinary 在媒体资源托管、CDN 分发、转换能力和后台管理上已经足够成熟，因此本项目更适合把精力放在“前台产品体验”和“后台安全管控”上，而不是重复建设底层媒体能力。

### 3.2 项目价值

- 对使用者：更快上传、更快复制链接、更少上下文切换。
- 对运营者：直接复用 Cloudinary 后台，减少系统复杂度。
- 对开发者：可先做最小可用版本，再按需要逐步增加权限、搜索、签名上传和日志能力。
- 对项目本身：保留从“个人图床”演进到“轻量媒体管理平台”的扩展空间。

## 四、目标、非目标与成功标准

### 4.1 项目目标

- 构建一个界面美观、操作直观、支持拖拽上传的在线图床页面。
- 上传后的图片可即时返回原图链接、Markdown 链接和 HTML 链接。
- 上传资源与 Cloudinary 官方 Media Library 保持同一后端资源池，实现天然同步。
- 增加管理员模式，支持对资源进行安全删除。
- 保留后续扩展空间，包括签名上传、标签分类、资源搜索、批量操作与访问控制。

### 4.2 非目标

本项目当前阶段明确不追求以下目标：

- 不构建复杂计费系统、租户系统、多角色细粒度权限系统。
- 不完全替代 Cloudinary 官方媒体库的高级资产管理能力。
- 不在首期提供桌面端客户端、浏览器插件或离线同步。
- 不在首期引入内容审核、社区互动或评论体系。

### 4.3 成功标准

| 指标 | 目标值 | 说明 |
|---|---:|---|
| 上传成功率 | >= 99% | 正常网络环境下，主流图片上传稳定完成 |
| 首屏可用时间 | < 2.5 秒 | Vercel 生产环境中首页应保持轻量 |
| 同步一致性 | 100% | Web 端上传成功后，Cloudinary 官方后台可查看同一资源 |
| 删除可靠性 | 100% | 管理员删除后，资源在 Cloudinary 中不可继续访问 |
| 易用性 | 高 | 新用户在 10 秒内能完成单图上传并复制链接 |
| 可维护性 | 中高 | 文档、目录结构与接口边界清晰，便于后续扩展 |

### 4.4 核心上线判断

满足以下条件即可认定 MVP 达标：

- 首页可上传图片。
- 上传成功后可立即复制不同格式的链接。
- 上传资源在 Cloudinary Media Library 中可见。
- 管理员登录后可删除资源。
- 删除后前台和资源链接状态保持一致。

## 五、目标用户与典型场景

### 5.1 目标用户

| 用户角色 | 画像 | 核心诉求 |
|---|---|---|
| 个人博主 | 经常写文章、发笔记、贴图 | 快速上传并复制 Markdown 图片链接 |
| 小团队内容运营 | 需要频繁分享素材链接 | 统一入口、统一资源池、减少后台切换 |
| 开发者本人/站长 | 希望低运维成本搭建可控图床 | 快速上线、可删除、后续可扩展 |

### 5.2 典型使用场景

1. 用户进入首页，拖拽一张图片上传。
2. 系统显示上传进度。
3. 上传成功后展示预览图与三种链接格式。
4. 用户点击复制 Markdown 链接并粘贴到文章中。
5. 管理员登录后台查看近期资源。
6. 管理员对误传、违规或过时图片执行删除。

### 5.3 核心用户故事

- 作为访客，我希望拖入图片后就能马上拿到链接，而不需要理解 Cloudinary 后台。
- 作为内容创作者，我希望复制 Markdown 链接时格式已经准备好，减少手动拼接。
- 作为管理员，我希望误传图片可以被安全删除，且删除过程不暴露密钥。
- 作为项目维护者，我希望首期版本简单、稳定，后续再逐步加入签名和审计能力。

## 六、项目范围

### 6.1 本期范围（In Scope）

- 首页上传页：拖拽上传、点击上传、进度反馈、成功/失败提示。
- 上传结果区：图片预览、原图 URL、Markdown、HTML 一键复制。
- 最近上传记录：优先展示当前会话的最近上传结果。
- 管理员登录：基于白名单邮箱或身份认证的后台入口。
- 管理员资源列表：展示近期资源，至少支持时间排序。
- 删除接口：管理员对指定资源执行安全删除。
- 基础品牌化 UI：保证页面具备明确的视觉风格，不做工具页式粗糙界面。

### 6.2 下一阶段范围（Near Scope）

- signed upload 替代完全开放的 unsigned upload。
- Cloudinary `folder` / `tag` 管理策略落地。
- 删除日志与管理员操作审计。
- 管理页搜索、筛选与分页。
- 批量操作和资源状态标记。

### 6.3 暂不纳入本期（Out of Scope）

- 复杂租户体系、多团队协作。
- 完整素材库替代方案。
- 图片审核、内容识别与违规自动处理。
- 多媒体类型统一管理平台。
- 移动 App、桌面 App、浏览器插件。

## 七、方案选型与总体架构

### 7.1 选型结论

| 层级 | 技术方案 | 选择理由 |
|---|---|---|
| 前端框架 | Next.js（建议 App Router） | 适合快速搭建产品站点，且天然适配 Vercel |
| 样式方案 | Tailwind CSS + 少量自定义组件 | 便于快速形成视觉风格与响应式布局 |
| 上传能力 | `next-cloudinary` 或 Cloudinary Upload Widget | 生态成熟，支持 direct upload 与后续 signed 模式 |
| 服务端能力 | Next.js Route Handlers / Server Actions | 便于承接删除、签名、鉴权等逻辑 |
| 存储与 CDN | Cloudinary | 承担媒体存储、处理、分发与后台管理 |
| 部署平台 | Vercel | 对 Next.js 友好，部署与环境变量配置简单 |
| 认证 | NextAuth 或 Clerk | 支撑白名单邮箱管理员访问控制 |

### 7.2 总体架构说明

1. 用户通过首页进入上传页面，选择本地图片。
2. 前端调用 Cloudinary 上传能力执行上传。
3. 图片直接进入 Cloudinary product environment，并出现在官方 Media Library。
4. 前端根据返回结果展示图片预览、`secure_url` 与其他格式链接。
5. 管理员登录后进入管理页，查看资源并触发删除操作。
6. 删除请求先进入自建服务端 API，再由服务端调用 Cloudinary 删除接口执行删除。

### 7.3 架构原则

- 上传可以前端直传，删除必须服务端执行。
- 任何涉及 `API_SECRET` 的能力都只应保留在服务端。
- 优先减少中间层，避免首期因额外数据库和同步机制增加复杂度。
- 在没有强需求前，不额外引入新依赖和新基础设施。
- 首期允许“无数据库”运行，但保留日志接入和审计扩展点。

### 7.4 逻辑架构图

```text
Browser
  -> Upload UI
  -> Cloudinary Upload
  -> Render preview / links

Admin Browser
  -> Auth Provider
  -> /admin
  -> /api/delete-image
  -> Cloudinary Admin/Delete API

Cloudinary
  -> Media storage
  -> CDN delivery
  -> Media Library
```

## 八、页面与信息架构

### 8.1 页面清单

| 页面/接口 | 是否首期 | 访问权限 | 目标 |
|---|---|---|---|
| `/` | 是 | 公开 | 上传图片、展示结果、复制链接 |
| `/admin` | 是 | 管理员 | 查看近期资源、删除资源 |
| `/login` 或认证弹窗 | 是 | 公开 | 管理员登录入口 |
| `/api/delete-image` | 是 | 管理员 | 服务端删除资源 |
| `/api/sign-cloudinary-params` | 否，可预留 | 管理员或受控访客 | 后续 signed upload |

### 8.2 首页信息架构

首页建议由以下模块组成：

- 品牌头部：项目名称、副标题、简短价值说明。
- 主上传区：拖拽区域、点击上传按钮、格式与大小限制说明。
- 上传状态区：上传中、成功、失败三种状态。
- 结果区：预览图、资源基本信息、复制按钮组。
- 最近上传区：展示当前会话最近上传的若干条记录。
- 页脚信息：Cloudinary / 部署说明、版权或维护信息。

### 8.3 管理页信息架构

- 顶部：管理员身份信息、退出登录。
- 列表区域：缩略图、文件名、上传时间、`public_id`、删除操作。
- 操作反馈：删除确认弹窗、删除成功/失败提示。
- 空状态：无资源、无权限、加载失败三类状态提示。

## 九、核心功能设计

### 9.1 访客端功能

| 模块 | 功能点 | 说明 |
|---|---|---|
| 上传区 | 拖拽 / 点击上传 | 支持常见图片格式，建议限制单文件大小与类型 |
| 上传区 | 文件校验 | 至少校验 MIME 类型、大小上限、数量上限 |
| 结果区 | 预览图 | 上传完成后即时展示缩略图或原图预览 |
| 结果区 | 链接复制 | 支持原图 URL、Markdown、HTML 三类格式 |
| 结果区 | 元信息展示 | 显示文件名、尺寸、资源类型、上传时间 |
| 体验优化 | 状态反馈 | 上传中、成功、失败都应给予清晰提示 |
| 品牌化 | 视觉风格 | 提供卡片式布局、留白、渐变背景与适度动效 |

### 9.2 管理员功能

| 模块 | 功能点 | 说明 |
|---|---|---|
| 认证 | 管理员登录 | 仅白名单邮箱或指定账号可进入 |
| 资源管理 | 列表查看 | 优先查看近期资源，后续可扩展搜索 |
| 资源管理 | 删除资源 | 调用服务端 API 删除指定 `public_id` |
| 审计 | 操作记录 | 建议保留删除日志，便于问题追踪 |
| 安全 | 最小权限 | 普通用户只允许上传，不允许删除 |

### 9.3 首期体验细节要求

- 上传区域在桌面端和移动端都应可用。
- 复制按钮点击后必须给出即时反馈。
- 上传失败时，错误提示不能只有“失败”，至少应提示大小、格式或网络错误。
- 删除操作必须要求二次确认。
- 删除成功后列表必须即时更新，避免用户误判状态。

## 十、上传、同步与删除机制

### 10.1 同步机制

由于项目直接使用 Cloudinary 的上传能力，上传后的资源不会先进入本地数据库或其他中间层，而是直接进入 Cloudinary 的资源环境。

因此，Web 端与 Cloudinary 官方后台不是“两套系统同步”，而是“同一资源池的两个入口”。这意味着只要上传成功，Cloudinary 媒体库即可查看该资源。

### 10.2 上传策略

首期建议采用以下策略：

- MVP 阶段允许使用 unsigned upload，以缩短上线时间。
- 必须限制格式、大小和单次上传数量，避免滥用。
- 必须统一 `folder` 与 `tag` 规则，避免后台资源混乱。
- 上传结果返回后立即构造链接格式，不再依赖额外接口转换。

### 10.3 删除机制

删除必须通过服务端执行。前端仅负责发起删除请求并传递 `public_id`；服务端完成管理员校验后，使用 Cloudinary 管理凭据调用删除接口。

此设计既满足功能需求，也符合密钥不出前端的安全原则。

**删除实现原则：**

- 不在浏览器中保存 Cloudinary `API Secret`。
- 不允许匿名删除。
- 删除成功后应更新前端状态并校验资源不可继续访问。
- 删除动作需要记录操作者、时间、资源标识和结果。

### 10.4 未来升级路径

- 从 unsigned upload 升级为 signed upload。
- 增加服务端上传参数签名与策略校验。
- 对来源、目录、标签和上传频率进行更细的限制。

## 十一、安全与权限设计

### 11.1 风险矩阵

| 风险项 | 风险说明 | 控制措施 |
|---|---|---|
| 匿名滥传 | 开放上传可能被恶意刷资源 | 限制文件大小、数量、格式；后续升级为 signed upload |
| 前端泄露密钥 | 若将 secret 暴露到客户端会导致资产被任意管理 | 所有管理接口只在服务端执行 |
| 误删 | 管理员操作可能删除错误资源 | 增加二次确认、删除日志与预览信息 |
| 未授权访问后台 | 非管理员进入管理页或调用接口 | 使用登录态与白名单邮箱校验 |
| 资源管理失控 | 不同来源资源混杂难以管理 | 使用 `folder`、`tag`、命名规范做隔离 |
| 链接滥用 | 大量外链导致资源消耗不可控 | 后续结合 Cloudinary 策略和监控做治理 |

### 11.2 权限策略

- **访客权限：** 仅上传与复制链接。
- **管理员权限：** 查看资源、删除资源、查看日志。
- **服务端接口：** 删除、签名、批量管理等接口均要求已认证管理员访问。
- **环境变量：** `NEXT_PUBLIC_*` 仅存公开信息；`API_KEY` 与 `API_SECRET` 仅在服务端保存。

### 11.3 白名单建议

- 使用邮箱白名单作为管理员最小可行权限模型。
- 白名单建议通过环境变量维护，而不是硬编码到页面逻辑中。
- 服务端必须再次校验管理员身份，不能只靠前端隐藏按钮。

## 十二、数据、命名与资源规范

### 12.1 Cloudinary 资源规范

- 建议使用 Cloudinary `folder` 对站点上传资源进行分层，例如 `cloudnest/uploads`。
- 建议在上传时追加固定标签，例如 `imagebed`、`mvp`、`web-upload`。
- 建议 `public_id` 采用“时间戳 + 随机串 + 规范化文件名”的格式，减少重名风险。

### 12.2 上传约束建议

| 项目 | 建议值 | 说明 |
|---|---|---|
| 支持格式 | `jpg` `jpeg` `png` `webp` `gif` | 首期优先图片格式 |
| 单文件大小 | 10 MB 以内 | 可根据使用场景调整 |
| 单次上传数量 | 1 到 5 张 | 首期以单图体验优先 |
| 默认目录 | `cloudnest/uploads` | 便于统一管理 |
| 默认标签 | `imagebed` `mvp` | 便于后台筛选 |

### 12.3 应用侧数据建议

首期可以不接数据库，但建议预留两类数据结构：

1. 最近上传记录
   - 当前会话可存浏览器本地状态。
   - 若后续需要跨设备查看，再接入数据库。

2. 删除审计记录
   - 建议至少包含：操作者、`public_id`、资源 URL、删除时间、删除结果、错误信息。

### 12.4 审计记录建议字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 唯一记录标识 |
| `actorEmail` | string | 管理员邮箱 |
| `publicId` | string | Cloudinary 资源标识 |
| `resourceType` | string | 资源类型 |
| `secureUrl` | string | 删除前资源链接 |
| `action` | string | 固定值 `delete` |
| `status` | string | `success` / `failed` |
| `message` | string | 结果说明或错误信息 |
| `createdAt` | datetime | 操作时间 |

## 十三、接口设计与契约建议

### 13.1 接口原则

- 能直传就不额外绕应用服务器，减少中间链路。
- 只有密钥相关与管理类能力才通过服务端 API。
- 接口响应格式应统一，便于前端处理提示与空状态。

### 13.2 删除接口

**接口路径：** `POST /api/delete-image`

**请求体示例：**

```json
{
  "publicId": "cloudnest/uploads/20260325_xxx_demo",
  "resourceType": "image",
  "invalidate": true
}
```

**成功响应示例：**

```json
{
  "success": true,
  "message": "Asset deleted successfully",
  "data": {
    "publicId": "cloudnest/uploads/20260325_xxx_demo",
    "result": "ok"
  }
}
```

**失败响应示例：**

```json
{
  "success": false,
  "message": "Unauthorized"
}
```

### 13.3 删除接口校验要求

- 必须校验登录态是否存在。
- 必须校验当前用户邮箱是否在管理员白名单中。
- 必须校验 `publicId` 非空且符合允许删除的目录前缀。
- 删除失败时返回明确错误信息，避免前端只能显示通用失败提示。

### 13.4 签名接口预留

**接口路径：** `POST /api/sign-cloudinary-params`

用途：

- 后续将 unsigned upload 升级为 signed upload 时使用。
- 对上传目录、标签、时间戳与允许参数做服务端签名与约束。

### 13.5 前端统一响应处理

前端处理接口时，统一使用以下逻辑：

- `success = true` 走成功提示与状态更新。
- `success = false` 显示明确错误文案。
- 网络异常、超时和 5xx 单独归类提示，便于排障。

## 十四、前端交互与视觉要求

### 14.1 视觉方向

页面不应呈现传统工具站的粗糙质感，建议具备以下视觉特征：

- 明确的品牌标题与副标题。
- 大面积留白与中心卡片布局。
- 柔和但有识别度的渐变或纹理背景。
- 上传区具备明显的拖拽反馈状态。
- 结果区具备结构层级，不只是原始文本堆叠。

### 14.2 关键交互状态

| 状态 | 页面表现 |
|---|---|
| 初始状态 | 显示拖拽区、说明文案和格式限制 |
| 上传中 | 展示进度条、禁用重复提交、显示当前文件名 |
| 上传成功 | 展示缩略图、链接块、复制成功提示 |
| 上传失败 | 展示可理解错误信息和重试动作 |
| 删除确认 | 弹窗展示缩略图、文件名、风险提示 |
| 删除成功 | 列表即时移除或更新状态 |
| 无权限 | 显示受限文案而不是空白页 |

### 14.3 移动端要求

- 上传按钮在移动端可点击且无遮挡。
- 结果卡片在窄屏下纵向排列，不出现复制按钮溢出。
- 管理页表格在移动端应退化为卡片列表，而不是保留不可读的横向表格。

- 验证上传后的资源能在 Cloudinary 官方后台查看。

- 优化首屏加载和图片展示体验。
- 完成基础异常提示和重试策略。

#### 阶段 3：管理员删除

- 接入认证方案，建立管理员白名单。

- 前端增加删除确认与删除结果反馈。
- 联调验证删除后资源不可访问。

#### 阶段 4：加固与扩展

- 增加日志与删除审计。

1. 先把首页上传主链路跑通。
2. 再完善结果区、复制与最近上传。

## 二十、风险与后续建议

本项目首要风险不在页面开发，而在开放上传后的资源治理与后台权限控制。

建议上线初期以简单清晰的策略快速落地：

- 第一优先级是完成“上传成功 + 官方后台可见 + 页面好看”。
- 第二优先级是完成管理员删除闭环。
- 第三优先级再做 signed upload、标签体系、操作日志与更完善的管理后台。

这样既可以尽快上线验证需求，又不会在初期陷入过度设计。

### 20.1 当前最主要风险

- unsigned upload 可能被滥用。
- 无数据库时，最近上传和审计能力有限。
- 管理后台如果没有二次校验，可能出现前端隐藏但接口可调用的问题。

### 20.2 对应建议

- 先通过限制大小、格式、数量降低滥用风险。
- 尽早统一目录与标签规范，避免 Cloudinary 后台污染。
- 尽快把删除操作沉淀为标准审计日志。
- 当上传量变大时，再切换到 signed upload 与更细的权限模型。

**推荐推进策略：** 先做可用，再做可管，最后做可扩展。项目首要目标是快速形成一个稳定、好看、可运营的图床入口。
